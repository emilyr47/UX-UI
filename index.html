About

Welcome to my Final Digital Project for CI 4312. Throughout this semester, I have taken a special interest in usability, and, specifically how human factors impact and interact with the web. Usability relates to how humans interact with user interfaces and the web. This project is centered around the heuristics of user interface (UI) design, written and developed by Jakob Nielsen. Control & Freedom

Users may often end up in areas of a website by mistake. They may click a link on accident, open a new window and wish to exit, or just wish to return to a previous page. While most web browsers provide a back button or a default escape route, UX/UI designers should never rely on this. First of all, while your code is internal and is great at communicating with itself, a web browser is an external application that simply opens your website. Because of this, a web browser may not always communication with your website perfectly and therefore, using the browser's back button may not get the user to where they want to go. Second, relying on a web browser to provide a user with control also requires the user to know their way around the web browser, which is not always a guarantee. Additionally, all web browsers behave slightly differently. So, even if your user knows how to use Google Chrome well, they may not be as familiar with Safari and that could be the only browser they have on their phone when they try to access your site. Finally, you also never know when a web browser is going under maintenance, changing their functionality, or eliminating or altering specific features. Having no control over this means you also have no control for your users if Internet Explorer decides to remove their back button. The example below shows an example of poor user control and freedom versus great user control and freedom.

Error Prevention

Error preventions sounds more technical than it actually is. While it is always important to create a user experience where errors are uncommon, in usability, error prevention does not necessarily mean preventing error. Instead, it means showing your users how fix errors that may pop up or come about when using your website. These can be simple things like a 404 error or an error message pop up. These are good examples because a user can always research a 404 error to determine its cause and then its solution and an error message pop up would usually give troubleshooting tips. The example below shows a form of poor error prevention and good error prevention.

Click the "Login" button in each example.

Website #1: No Error Prevention

We'll never share your email with anyone else.
Login

  • No error message appears
  • The user would need to speculate as to why the login is not working (the login button is broken, the webpage doesn't work, my internet is down, etc.)

Website #2: Good Error Prevention
We'll never share your email with anyone else.
Login

We do not recognize your username and/or password
Please check your login information and try again.

  • Provides a diagnosis of why the login is not working
  • Makes a suggestion for the user to try troubleshooting
  • Puts the error message in plain language for the basic user to understand






Aesthetic & Minimalistic Design

Today's web trends feature very minimalistic designs. Information that is lengthy and unnecessary should never be included on a website. A website should always be scannable and always feel as though it had "white space to spare." Loading your website with too much content and too much text can easily overwhelm a user and make it difficult for them to navigate your website. Recognize & Diagnose

While we have already explained what it means to help users out with errors, many developers and designers still do not accomplish this aspect of usability on their websites. The main cause of this is error codes. While error codes can be helpful for identifying problems in certain cases, expecting a user to solve a problem with just an error code is unrealistic. Most error codes these days are easy enough to research and diagnose, but this requires research on the user's part, which is not their job. Additionally, this requires the user to know how to research the particular code, find an appropriate solution that they understand, and then implement that solution. This is far too many steps to assume of your users. So, does this mean no error codes at all? Not necessarily. In many cases, error codes are very helpful, and they can be continued to be used as a diagnostic tool. However, additional information written in plain language should always supplement the error code. Additional descriptions help your user to recognize specific problems and then diagnose them so that they can recover from these errors. The examples below show both a poor error and a good error.

Error #1: Poor Error

  • Error code is listed
  • No description
  • No next steps or help in diagnosis
  • No way to return to previous pages or view more options

Error #2: Good Error

  • Features an error code
  • Includes a short description explaining what the error code means
  • Specifies potential next steps for diagnosis and recovery
  • Provides additional links that may direct the user to the correct place
