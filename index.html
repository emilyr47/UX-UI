About

Welcome to my Final Digital Project for CI 4312. Throughout this semester, I have taken a special interest in usability, and, specifically how human factors impact and interact with the web. This project is centered around the ten heuristics of user interface (UI) design, written and developed by Jakob Nielsen. Each of these ten heuristics plays an important role in ensuring accessibility and usability on the web. In this project, I will demonstrate each of these ten heuristics by explaining them and then providing applicable examples of how they can be exercised in UI design and can imporve user experience (UX).



Visibility

Visibility informs users of system status by providing feedback. This could look like a spinning loader as a program loads, a progress bar gradually filling during an update, or any visual or form of feedback that tells the user that the request is working. A simple example of this is shown in the buttons below.

  • No color change in button on hover
  • No text indicating where the button is going or if it is going to where the user wants
  • No link appears on hover

Real World

Websites should always use plain language and "real world" terms. Users should be familiar with the language on a website and should not be required to learn technical jargon in order to navigate it. While a website may feature technical terms or concepts, (e.g. online medical journals, privacy policies, terms and aggreements, etc.) this is just the website's content. These technical terms should not also be used in navigation, functionality, and usability.



Control & Freedom

Users may often end up in areas of a website by mistake. They may click a link on accident, open a new window and wish to exit, or just wish to return to a previous page. While most web browsers provide a back button or a default escape route, UX/UI designers should never rely on this. First of all, while your code is internal and is great at communicatinhg with itself, a web browser is an external application that simply opens your website. Because of this, a web browser may not always communication with your website perfectly and therefore, using the browser's back button may not get the user to where they want to go. Second, relying on a web browser to provide a user with control also requires the user to know their way around the web browser, which is not always a guarantee. Additionally, all web browsers behave slightly differently. So, even if your user knows how to use Google Chrome well, they may not be as familiar with Safari and that could be the only browser they have on their phone when they try to access your site. Finally, you also never know when a web browser is going under maintenance, changing their functionality, or eliminating or altering specific features. Having no control over this means you also have no control for your users if Internet Explorer decides to remove their back button. The example below shows an example of poor user control and freedom versus great user control and freedom.

  • No way to exit the pop-up
